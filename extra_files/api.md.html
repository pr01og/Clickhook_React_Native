API Documentation (Internal Use)

Base URL: https://leads.clickhook.io/api/

General

ClickHook’s API is pretty consistent throughout endpoints so some things are applicable for every endpoint. For example:

Errors

When there’s an error, ClickHook will try its best to return the exact error.

  • In case there’s a server error, you will see a 500 response.

  • In case of a validation error, you will get back a 422 response with an object like this:

  {
    errors: {
      property: Array[]
    }
  }

It will return the errors array for every property that was invalid.

  • In case of an unauthorized request, it will return a 401 response — this is usually when the user hasn’t logged in and you’re trying to fetch something private

  • In case of a forbidden request — i.e. a user has logged in but is trying to access something he does not have access to — it will return a 403 response.

REST

ClickHook’s API generally follows a conventional REST API. So, for just about every endpoint, the convention would be:

Method Endpoint Remark
GET /resources Fetch a list of all resources
POST /resources Create a new resource
GET /resources/:id Fetch the resource with that ID
PUT /resources/:id Update the resource with that ID
DELETE /resources/:id Delete the resource with that ID

Authentication

endpoint: /token method: POST

Request: grant_type: Grant type can either be password or refresh_token.

If grant_type is password, you can send these:

username
password

If grant_type is “refresh_token”, you can send these:

refresh_token

When you authenticate for the first time, you send a JSON object like so:

  {
    "grant_type": "password",
    "username",
    "password"
  }

If there is an error, you will get back a 422 with a response like:

  {
    "message": "The e-mail or password you entered is wrong."
  }

Otherwise, you will get a 201 CREATED with:

  {
    "token_type": "bearer",
    "access_token",
    "refresh_token",
    "expires_in": 10800
  }

Essentially, you get an access_token that you can use to authorize your requests, a refresh token, and an expiry time. By default, the access_token expires in 3 hours so you need to send a refresh request with the refresh token to get a new access_token. Something like:

  {
    "token_type": "refresh_token",
    "access_token"
  }

This will return the same 201 CREATED with a new token.

The access_token that you get needs to be sent as an Authorization header like so:

  Authorization: `Bearer ${accessToken}`

This will ensure you authenticate as the right user.

Finding the current user

endpoint: /users/me method: GET

Response:

{
  user: {
    id,
    name,
    email,
    role,
    username,
    timezone
    },
  accounts: [{
    id,
    name
    }],
  users: []
}

This endpoint returns a JSON object with the current user, the current user’s account, and an array of other users in the current user’s account.

Finding any other user

endpoint: /users/:id method: GET

Response:

{
  user: {
    id,
    name,
    email,
    role,
    username,
    timezone
    }
}

Similar to the current user endpoint, except it returns only the user whose ID you provided.

Finding leads

endpoint: /leads method: GET query: { page: Integer, search: String, tags: Array, sources: Array }

Query parameters allow you to filter data using specific fields. So, for example, if you want to return only those leads who match a particular search term, you should send a request like:

/leads?search="Dave"

Similarly, if you want to filter by tags, you send a request like:

/leads?tags[]="google"

Response:

{
  leads: [{
    id,
    name,
    email,
    company,
    city,
    state,
    address,
    zip,
    source,
    phone,
    email_hash,
    tags: [],
    custom_fields: {}
    }],
  instances: [{
    id,
    assigned_at,
    assigned_to_name,
    assigned_to_id,
    is_closed,
    is_lost,
    closed_at,
    lost_at,
    created_at,
    stage_id,
    stage_name,
    workflow_id,
    workflow_name
  }],
  meta: {
    total_count: Number
  }
}

Leads endpoint returns an array of leads (and their corresponding instances) in a normalized fashion.

Finding a single lead

endpoint: /leads/:id method: GET

Response:

{
  lead: {
    id,
    name,
    email,
    company,
    city,
    state,
    address,
    zip,
    email_hash,
    source,
    phone,
    tags: []
    custom_fields: {}
    },
 
    instances: []
  }

Lead endpoint returns a single lead and an array of instances associated with it.

Creating a lead

endpoint: /leads/ method: POST

Request:

{
  lead: {
    name,
    email: Required,
    phone: Required,
    source,
    company,
    city,
    state,
    address,
    zip,
    tags: []
    custom_fields: {}
    }
  }

Response:

{
  lead: {
    id,
    name,
    email,
    phone,
    source,
    company,
    city,
    state,
    address,
    zip,
    tags: []
    email_hash,
    custom_fields: {}
    },
 
    instances: []
  }

To create a lead, you send a POST request with (at least) the required fields. If the lead is created, you will get a 201 CREATED with the response outline above. In case of an error, the response will be a standard 422 response as outlined at the top of the document.

Updating a lead

endpoint: /leads/:id method: PUT

Request:

{
  lead: {
    company: "Google"
    }
  }

Response:

{
  lead: {
    id,
    name,
    email,
    phone,
    source,
    company: "Google",
    city,
    state,
    address,
    zip,
    tags: []
    email_hash,
    custom_fields: {}
    },
 
    instances: []
  }

Similar to how it’s outlined at the top of the document, you can send a list of the updated fields and you will get a 200 OK response with the lead, updated.
